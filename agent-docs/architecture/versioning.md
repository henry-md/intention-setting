## Extension Versioning

When shipping a new Chrome extension version, edit these source version values together:

- `extension/public/manifest.json` `version`. Chrome reads this loaded manifest for the installed extension version.
- `extension/package.json` `version` so build/package metadata matches.
- `public-site/lib/extensionVersion.ts` `EXTENSION_LATEST_VERSION`, which controls when the upgrade modal appears for users behind latest.
- `public-site/lib/extensionVersion.ts` `EXTENSION_MIN_SUPPORTED_VERSION`, which controls when the modal says the upgrade is required.

Regenerate derived artifacts after changing source versions:

- `extension/build/manifest.json` is generated by `cd /Users/Henry/Developer/intention-setting/extension && pnpm build`.
- `extension/extension.zip` is generated by `cd /Users/Henry/Developer/intention-setting/extension && pnpm zip`.

Chrome Web Store reads the uploaded zip's `manifest.json`; there is no separate manual version field there.

Deploy the public site after changing `public-site/lib/extensionVersion.ts`, or the API will keep serving the old latest/min-supported versions.

For normal upgrade-modal copy/tests, set `UPGRADE_MSG` from an empty string to the message text. A non-empty `UPGRADE_MSG` advertises the next patch version while keeping the minimum supported version unchanged, so current users see the standard non-required upgrade modal with that copy. Use `FORCE_UPGRADE_MODAL_TO_SHOW=true` only as a local escape hatch for unthrottled required-modal testing; keep it false in production. The extension under test must also be pointed at the local public site (`VITE_PUBLIC_SITE_URL=http://localhost:4232`). If testing from `extension/build`, run `pnpm build:local` in `extension`; ordinary `pnpm build` uses `.env.production` and calls the Railway API, so it will not see the local force flag. HMR will not update the version shown in `chrome://extensions`; reload the extension card after changing `manifest.json`.
